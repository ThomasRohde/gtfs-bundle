# Processes

This index is a progressive-disclosure entry point for practical workflows that use the GTFS Schedule catalog against real feed data.

| concept | type | description |
|---|---|---|
| [find and use gtfs feed](find-and-use-gtfs-feed.md) | Process | Practical workflow for moving from a place-based timetable question to a concrete GTFS feed query. |
| [validate a gtfs feed](validate-a-gtfs-feed.md) | Process | Workflow for checking a downloaded GTFS Schedule feed before using it as evidence. |
| [query stop departures](query-stop-departures.md) | Process | Repeatable GTFS Schedule workflow for departure lists at an exact stop or station area. |
| [build active service calendar](build-active-service-calendar.md) | Process | Workflow for deriving active service IDs for a requested GTFS service date. |
| [resolve station area](resolve-station-area.md) | Process | Workflow for deciding whether a GTFS stop question refers to one stop, a parent station, platforms, or nearby stop points. |
| [decide static vs realtime source](decide-static-vs-realtime-source.md) | Process | Decision workflow for choosing GTFS Schedule, GTFS Realtime, or a journey-planner API for a transit question. |
| [check gtfs spec freshness](check-gtfs-spec-freshness.md) | Process | Workflow for refreshing claims about GTFS Schedule, Realtime, and extension maturity before answering. |
| [interpret fares v2](interpret-fares-v2.md) | Process | Workflow for determining whether and how a GTFS feed can answer rider-facing fare questions. |
| [query demand responsive service](query-demand-responsive-service.md) | Process | Workflow for interpreting GTFS Flex and on-demand service information in a static Schedule feed. |
| [inspect station accessibility](inspect-station-accessibility.md) | Process | Workflow for answering static GTFS questions about wheelchair access, pathways, levels, and station navigation. |
